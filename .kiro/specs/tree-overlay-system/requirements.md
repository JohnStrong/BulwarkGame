# Requirements Document

## Introduction

The tree-overlay-system separates tree tile rendering into two independent layers: a ground layer (grass sprite) and an overlay layer (a new transparent-background tree sprite). Currently, tree sprites bake the grass base and tree canopy into a single 64×32 diamond tile. The new approach draws the grass ground tile first, then composites a taller, transparent-background tree sprite on top — centered on the tile but allowed to bleed upward into the tile above — giving trees visual height and depth consistent with how unit sprites are rendered today.

The feature touches four areas of the codebase:
1. **Sprite generator** — new functions to produce tree overlay sprites with transparent backgrounds
2. **Sprite constants** — new overlay sprite name entries
3. **Level loader** — tree tile objects gain an `overlay` field alongside the existing `sprite` field
4. **Renderer** — `IsoRenderer.drawTerrain` draws the ground sprite first, then the overlay sprite offset upward

Existing `tree-1` through `tree-7` sprites (grass + tree baked together) are kept for backward compatibility.

## Glossary

- **Tile**: A single cell in the isometric hex grid, described by `{ row, col, x, y, sprite, overlay? }`.
- **Ground_Sprite**: The 64×32 flat isometric diamond sprite drawn at the tile's base position (e.g. `grass-short-1`).
- **Overlay_Sprite**: A taller transparent-background sprite (e.g. 64×48 or 64×64) drawn on top of the ground sprite, offset upward so the tree appears to stand on the tile.
- **Tree_Overlay_Generator**: The new functions inside `generate-iso-sprites-br-tl.js` that produce overlay sprites with alpha=0 outside the tree shape.
- **Level_Loader**: `js/game-logic/level-loader.js` — parses level `.txt` files and produces tile objects.
- **IsoRenderer**: `js/game-logic/lib/iso-renderer.js` — draws terrain tiles and units to the canvas.
- **SpriteManager**: `js/game-logic/sprites.js` — loads and draws sprite images; owns `spriteList`.
- **Sprite_Constants**: `js/level-generators/lib/sprite-constants.js` — canonical sprite name registry.
- **Build_Pipeline**: `js/level-generators/build-sprites.js` — orchestrates all sprite generators and packs the atlas.
- **Atlas**: The packed sprite sheet (`assets/sprites/atlas-0.png` + `atlas.json`) consumed at runtime.
- **Oak**: Level character `O` — maps to oak tree variants.
- **Pine**: Level character `P` — maps to pine tree variants.
- **Shrub**: Level character `S` — maps to shrub variants.
- **Overlay_Offset**: The number of pixels the overlay sprite is shifted upward relative to the tile's top-left corner, so the tree base aligns with the ground tile.

## Requirements

### Requirement 1: Tree Overlay Sprite Generation

**User Story:** As a developer, I want new tree overlay sprites with transparent backgrounds generated by the sprite pipeline, so that trees can be rendered as a separate layer on top of grass tiles.

#### Acceptance Criteria

1. THE Tree_Overlay_Generator SHALL produce overlay sprites for each tree variant: `tree-oak-overlay-1`, `tree-oak-overlay-2`, `tree-oak-overlay-3`, `tree-pine-overlay-1`, `tree-pine-overlay-2`, `tree-shrub-overlay-1`, `tree-shrub-overlay-2`.
2. WHEN a tree overlay sprite is generated, THE Tree_Overlay_Generator SHALL set alpha=0 for every pixel outside the tree shape (trunk and canopy), matching the transparent-background approach used by unit sprites.
3. WHEN a tree overlay sprite is generated, THE Tree_Overlay_Generator SHALL produce a canvas that is 64 pixels wide and at least 48 pixels tall, allowing the tree to bleed upward beyond the 32-pixel tile height.
4. WHEN a tree overlay sprite is generated, THE Tree_Overlay_Generator SHALL draw the tree trunk and canopy using the same palette colors and layered-canopy technique as the existing `generateTree` function, adapted for a transparent background.
5. THE Tree_Overlay_Generator SHALL produce visually distinct sprites for oak (rounded canopy), pine (pointed/conical canopy), and shrub (low, wide canopy) variants.
6. WHEN the sprite build pipeline runs, THE Build_Pipeline SHALL invoke the Tree_Overlay_Generator and include all seven overlay sprites in the atlas.

### Requirement 2: Sprite Name Registration

**User Story:** As a developer, I want the new overlay sprite names registered in the sprite constants and SpriteManager, so that the runtime can load and draw them.

#### Acceptance Criteria

1. THE Sprite_Constants SHALL define a `TREE_OVERLAY_SPRITES` registry object containing the canonical names for all seven tree overlay sprites.
2. THE SpriteManager.spriteList SHALL include all seven tree overlay sprite names so they are loaded during `loadAll()` and packed into the atlas.
3. WHEN the atlas is built, THE Build_Pipeline SHALL read the overlay sprite names from `TREE_OVERLAY_SPRITES` and include them in the sprite entries passed to `packAtlas()`.
4. IF an overlay sprite PNG is missing at atlas-pack time, THEN THE Build_Pipeline SHALL log a structured diagnostic error and exit with a non-zero code.

### Requirement 3: Level Loader Tile Data

**User Story:** As a developer, I want tree tiles produced by the level loader to carry both a ground sprite and an overlay sprite name, so that the renderer knows what to draw for each layer.

#### Acceptance Criteria

1. WHEN the Level_Loader parses an `O` character, THE Level_Loader SHALL produce a tile with `sprite` set to `grass-short-1` or `grass-short-2` (selected by `tileHash`) and `overlay` set to any valid tree overlay sprite name from `TREE_OVERLAY_SPRITES` (selected by `tileHash`).
2. WHEN the Level_Loader parses a `P` character, THE Level_Loader SHALL produce a tile with `sprite` set to `grass-short-1` or `grass-short-2` and `overlay` set to any valid tree overlay sprite name from `TREE_OVERLAY_SPRITES` (selected by `tileHash`).
3. WHEN the Level_Loader parses an `S` character, THE Level_Loader SHALL produce a tile with `sprite` set to `grass-short-1` or `grass-short-2` and `overlay` set to any valid tree overlay sprite name from `TREE_OVERLAY_SPRITES` (selected by `tileHash`).
4. WHEN the Level_Loader parses any non-tree character, THE Level_Loader SHALL produce a tile with no `overlay` field.
5. THE Level_Loader SHALL preserve the existing `tileHash` function without modification so that tile variant selection remains deterministic.

### Requirement 4: Two-Layer Terrain Rendering

**User Story:** As a player, I want tree tiles to display a grass ground layer with a tree standing on top of it, so that the map looks visually richer and trees have proper depth.

#### Acceptance Criteria

1. WHEN IsoRenderer.drawTerrain iterates a tile that has an `overlay` field, THE IsoRenderer SHALL first draw `tile.sprite` (the ground sprite) at the tile's screen position using the standard tile dimensions (camera.tileW × camera.tileH).
2. WHEN IsoRenderer.drawTerrain iterates a tile that has an `overlay` field, THE IsoRenderer SHALL then draw `tile.overlay` centered horizontally on the tile and offset upward by the Overlay_Offset so the tree base aligns with the ground tile surface.
3. WHEN IsoRenderer.drawTerrain iterates a tile that has no `overlay` field, THE IsoRenderer SHALL draw only `tile.sprite` using the existing code path, preserving backward compatibility.
4. WHEN a tile is selected or hovered, THE IsoRenderer SHALL draw the diamond outline after both the ground sprite and the overlay sprite have been drawn for that tile.
5. THE IsoRenderer SHALL draw overlay sprites using SpriteManager.draw with the overlay sprite's native dimensions (64×48 or 64×64) rather than scaling to camera.tileW × camera.tileH.

### Requirement 5: Overlay Vertical Positioning

**User Story:** As a developer, I want the overlay sprite's vertical offset to be defined as a named constant, so that the tree base consistently aligns with the ground tile across all zoom levels.

#### Acceptance Criteria

1. THE IsoRenderer SHALL define a named constant `TREE_OVERLAY_OFFSET_Y` (in pixels, relative to the tile's top-left corner) used when positioning every overlay sprite.
2. WHEN the overlay sprite is drawn, THE IsoRenderer SHALL compute the draw Y position as `tileTopY - (overlayHeight - camera.tileH) + TREE_OVERLAY_OFFSET_Y`, ensuring the bottom of the overlay sprite aligns with the bottom of the ground tile.
3. WHEN the overlay sprite is drawn, THE IsoRenderer SHALL center it horizontally: draw X = `tileCenterX - overlayWidth / 2`.

### Requirement 6: Backward Compatibility

**User Story:** As a developer, I want the existing tree-1 through tree-7 sprites and their level-loader mappings to remain functional, so that no existing level data is broken by this change.

#### Acceptance Criteria

1. THE Sprite_Constants SHALL retain all existing entries in `TERRAIN_SPRITES` for `tree-1` through `tree-7`.
2. THE SpriteManager.spriteList SHALL retain `tree-1` through `tree-7`.
3. THE Build_Pipeline SHALL continue to generate `tree-1` through `tree-7` sprites via the existing `generateTree` function.
4. IF a tile object has no `overlay` field, THEN THE IsoRenderer SHALL draw only `tile.sprite` at the standard tile dimensions (camera.tileW × camera.tileH) and SHALL NOT draw any overlay.

### Requirement 7: Atlas Build Integration

**User Story:** As a developer, I want the build pipeline to automatically include the new overlay sprites in the atlas, so that no manual steps are needed after running the build.

#### Acceptance Criteria

1. WHEN the Build_Pipeline runs, THE Build_Pipeline SHALL execute the Tree_Overlay_Generator as part of the existing generator sequence before the atlas-packing step.
2. THE Build_Pipeline SHALL read overlay sprite names from `TREE_OVERLAY_SPRITES` in `sprite-constants.js` and add them to the sprite entries array alongside terrain, castle, unit, enemy, and damaged-castle sprites.
3. WHEN the atlas is packed, THE Build_Pipeline SHALL verify that all seven overlay sprite PNGs exist in `OUTPUT_DIR` before calling `packAtlas()`.
4. THE Build_Pipeline SHALL verify that the final atlas file size remains below 4 MB after adding the overlay sprites, where the verification passes if and only if the actual file size is below the 4 MB limit.
